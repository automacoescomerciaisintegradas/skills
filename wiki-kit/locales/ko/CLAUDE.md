# wiki-kit 운영 스키마

이 디렉토리는 LLM이 지속적으로 업데이트하는 마크다운 기반의 지식기지입니다. 매번 원문을 검색하는 대신, 목표는 `wiki/`에 지식을 축적하면서 요약, 개념 지도, 상호참조, 모순 추적을 시간에 따라 성장시키는 것입니다.

## 1. 당신의 역할

- 이 wiki 프로젝트 루트의 관리자입니다
- `raw/`는 원문 계층이므로 읽을 수는 있지만 절대 수정해서는 안 됩니다
- `wiki/`는 당신이 관리하는 지식기지입니다
- `templates/`는 페이지 구조 참고 자료를 제공하므로 필요시 참고하되 보통 편집하지 않습니다
- 인간이 담당하는 것: 자료 입수, 우선순위 설정, 의사결정
- 당신이 담당하는 것: 요약, 정리, 링크 연결, 차이점 통합, 유지보수

## 2. 디렉토리 의미론

- `raw/sources/`: 원문 - 기사, 논문, PDF, 노트, 회의 자료, CSV 등
- `raw/assets/`: 이미지, 다이어그램, 첨부 파일
- `wiki/overview.md`: 이 wiki의 주제, 목적, 가설, 관점을 정리하는 상위 페이지
- `wiki/index.md`: 전체 wiki의 목차, 내용 기반 색인
- `wiki/log.md`: 입수, 질의, 린트의 시간순 로그
- `wiki/open-questions.md`: 미해결 질문, 연구 대상, 연기된 문제
- `wiki/sources/`: 각 원문 당 하나의 요약 및 평가 페이지
- `wiki/concepts/`: 개념, 주제, 이슈, 논쟁 페이지
- `wiki/entities/`: 사람, 회사, 제품, 단체, 시스템 등을 다루는 페이지
- `wiki/syntheses/`: 비교, 분석, 결론, 보고서 - 재사용성이 높은 질의 결과
- `wiki/maintenance/lint-reports/`: 정기적인 검토 보고서

## 3. 절대 규칙

1. `raw/`의 파일은 절대 이동, 편집, 삭제, 덮어쓰기하지 않습니다
2. 추측과 사실을 혼동하지 않습니다; 항상 증거의 강도를 표시합니다
3. 모든 내용은 한국어로 작성합니다
4. 모든 것을 마크다운으로 관리합니다
5. 새로운 지식을 추가할 때는 관련 페이지에 반영하는 것을 우선합니다
6. 모든 변경 시 `wiki/index.md`와 `wiki/log.md`를 업데이트합니다
7. 기존 페이지에 추가할 수 있으면 불필요하게 새 페이지를 만들지 않습니다
8. 모순을 발견했을 때 무언가를 무시하고 덮어쓰지 않습니다 - 어느 원문이 뭐라고 했는지 기록합니다
9. 인용은 필요한 최소한만 유지합니다; 긴 원문 그대로의 발췌는 피합니다
10. 확실하지 않을 때는 먼저 `index.md`와 관련 페이지를 읽어서 중복과 네이밍 충돌을 확인합니다

## 4. 언어 및 네이밍 규칙

- 본문은 한국어로 작성합니다
- 파일 이름은 원칙적으로 ASCII `kebab-case`를 사용합니다
- 표시 이름은 파일명이 아니라 본문 제목과 frontmatter의 `title`로 표현합니다
- 원문 요약의 권장 파일명: `YYYY-MM-DD_source-<slug>.md`
- 종합 분석의 권장 파일명: `YYYY-MM-DD_<topic-slug>.md`
- 엔티티와 개념 페이지는 오래 유지되므로 이름을 짧고 안정적으로 유지합니다

## 5. Frontmatter 규칙

새 페이지는 보통 다음 frontmatter를 포함해야 합니다:

```yaml
---
title: ""
type: source | concept | entity | synthesis | maintenance
status: draft | active | superseded
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: []
source_files: []
related_pages: []
---
```

참고 사항:

- `source_files`는 `raw/` 아래 실제 파일 경로를 저장소 루트 기준 상대 경로로 나열합니다 (예: `raw/sources/example.pdf`)
- `related_pages`는 관련 `wiki/` 페이지로의 상대 경로를 나열합니다
- `status`는 보통 `active`입니다; `superseded`는 오래된 결론이 대체되었을 때만 사용합니다
- `overview.md`, `index.md`, `log.md`, `open-questions.md` 같은 기본 스캐폴드 페이지도 `type: maintenance`를 사용합니다

## 6. 페이지 유형별 예상 내용

### source

- 자료의 요약
- 핵심 포인트
- 주장 및 증거
- 기존 wiki에 미치는 영향
- 미해결 질문

### concept

- 개념의 정의
- 현재 이해도
- 경쟁하는 견해와 논쟁
- 관련 자료 및 엔티티
- 향후 연구 방향

### entity

- 대상의 개요
- 핵심 사실
- 타임라인과 진화
- 다른 개념 및 엔티티와의 관계
- 모니터링할 포인트

### synthesis

- 질문
- 결론
- 증거로 사용한 페이지
- 불확실성
- 다음 조치

### maintenance

- 검토 범위
- 발견된 문제
- 권장 조치
- 우선순위 수준

## 7. 입수 절차

새 자료를 처리할 때는 항상 다음 순서를 따릅니다:

1. `wiki/index.md`와 `wiki/log.md`를 읽어서 최근 작업을 파악합니다
2. `raw/sources/`에서 대상 자료를 읽습니다
3. `templates/source-summary-template.md`를 참고하여 `wiki/sources/`에 요약 페이지를 만듭니다
4. 기존 `concepts/`, `entities/`, `overview.md`가 업데이트되어야 하는지 확인합니다
5. 중대한 모순이나 새로운 문제가 생기면 `open-questions.md`에 반영합니다
6. 새 페이지를 `wiki/index.md`에 추가하고 요약 및 날짜를 업데이트합니다
7. `wiki/log.md`에 입수 로그 항목을 추가합니다

### 입수 중 의사결정 기준

- 기존 개념을 강화하면 해당 개념 페이지에 추가합니다
- 새로운 고유명사가 나타나고 반복될 가능성이 있으면 엔티티 페이지를 만듭니다
- 주제가 중요하지만 아직 미숙하면 개념 페이지를 만들고 불확실한 부분을 표시합니다
- 한 번만 나오는 노트라면 원문 요약만으로 충분할 수 있습니다

## 8. 질의 절차

질문에 답할 때 대화 이력보다 wiki를 먼저 참고합니다.

1. `wiki/index.md`를 먼저 읽어서 후보 페이지를 파악합니다
2. 후보 페이지를 읽고, 필요시 원문 요약으로 돌아갑니다
3. 증거 강도 순서대로 정보를 정리하고 답합니다
4. 어떤 위키 페이지를 근거로 삼았는지 명시하고, 가능하면 응답 안에서 링크한다
5. 응답에서 사실과 해석을 구분합니다
6. 응답이 재사용 가능한 비교, 분석, 요약이면 `wiki/syntheses/`에 저장합니다
7. 저장했으면 `index.md`와 `log.md`를 업데이트합니다

### 종합 분석으로 저장할 가치가 있는 내용

- 비교 표
- 의사결정을 위한 정리된 자료
- 장문 분석
- 여러 원문에 걸친 결론
- 재사용될 가능성이 높은 FAQ 스타일 답변

## 9. 린트 절차

정기적인 검토 중에는 항상 다음을 확인합니다:

- 여러 페이지에 걸쳐 지속되는 모순된 진술이 있는가?
- 새 자료로 인해 오래된 결론이 구식이 되었는가?
- 고아 페이지가 쌓이고 있는가?
- 중요한 개념들이 원문 요약에 흩어져 있는가? (개념 페이지로 승격되지 않은 상태)
- 엔티티, 개념, 종합 분석 페이지로 승격되지 않은 중요 이슈가 있는가?
- 질문들이 `open-questions.md`에 미처리된 상태로 쌓이고 있는가?

린트 결과를 `wiki/maintenance/lint-reports/`에 저장하고 필요시 `open-questions.md`와 `index.md`에 반영합니다.

## 10. index.md 업데이트 규칙

- 각 페이지 항목은 한 줄로 요지를 전달합니다
- 카테고리별로 정리합니다
- 새 페이지를 만들 때마다 항목을 추가합니다
- 페이지 상태가 `superseded`가 되면 명시적으로 표기합니다
- 가능하면 업데이트 날짜를 포함합니다

권장 형식:

```text
- [페이지-제목](./path/to/page.md): 이 페이지가 다루는 내용을 한 문장으로 설명합니다. 마지막 업데이트: 2026-04-07
```

## 11. log.md 업데이트 규칙

로그는 추가 전용의 시간순 기록입니다. 기존 로그 항목을 재작성하지 않습니다.

다음 제목 형식을 일관되게 사용합니다:

```text
## [YYYY-MM-DD] ingest | 자료명
## [YYYY-MM-DD] query | 질문 요약
## [YYYY-MM-DD] lint | 범위
## [YYYY-MM-DD] update | 설명
```

각 로그 항목에는 최소한 다음을 포함해야 합니다:

- 무엇을 했는가
- 어떤 페이지를 만들거나 업데이트했는가
- 미해결 사항이 무엇인가

## 12. 상호 참조

- 상대 경로 마크다운 링크를 사용합니다
- 가능하면 양방향 링크를 만듭니다
- 원문 요약에서 개념/엔티티 페이지로, 그 반대로 링크합니다
- 단순히 관련 용어를 나열하지 말고 링크가 왜 관련이 있는지 맥락을 제공합니다

## 13. 작문 톤

- 간결하게 작성합니다
- 사실, 해석, 가설을 분리합니다
- 증거로 주장을 뒷받침합니다
- 불확실한 내용은 "미확인", "가설", "원문마다 다름" 같은 표현으로 명시적으로 표시합니다
- 대화 톤이 아닌 재사용을 위한 참고 자료 스타일로 작성합니다

## 14. 확실하지 않을 때의 우선순위

1. `raw/`는 건드리지 않습니다
2. `index.md`와 기존 페이지를 확인해서 중복을 피합니다
3. 원문 요약을 만듭니다
4. 개념/엔티티/종합 분석으로 최소한만 승격합니다
5. `index.md`와 `log.md`를 업데이트합니다

## 15. 참고 템플릿

새 페이지를 만들 때는 필요시 다음을 참고합니다:

- `templates/source-summary-template.md`
- `templates/concept-template.md`
- `templates/entity-template.md`
- `templates/synthesis-template.md`
- `templates/lint-report-template.md`
